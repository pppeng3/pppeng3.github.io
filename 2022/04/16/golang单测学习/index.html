一、Go单元测试基础

单元测试,通常是单独测试一个方法、类或函数,让开发者确信自己的代码能按预期运行,为确保代码可以测试且测试易于维护。

Go测试文件命名必须为_test.go,测试函数必须为Test*。
*_test.go文件中有三种类型的函数，单元测试函数、基准测试函数和示例函数:

类型 格式 作用
测试函数 函数名前缀为Test 测试程序的一些逻辑行为是否正确
基准函数 函数名前缀为Benchmark 测试函数的性能
示例函数 函数名前缀为Example 为文档提供示例文档

go test -v: 输出完整的测试结果
go test -run=*: -run参数对应一个正则表达式,只有函数名匹配上的测试函数才会被go test命令执行
go test -short: 不会执行耗时测试用例
go test -cover: 测试覆盖率,提供额外的-coverprofile=c.out用来将覆盖率相关的记录信息输出到一个文件,然后执行go tool cover -html=c.out,使用cover工具来处理生成的记录信息,该命令会打开本地的浏览器窗口生成一个Html报告。

mockgen命令用来为给定一个包含要mock的接口的go源文件,生成mock类源代码,mockgen有两种操作模式:源码(source)模式和反射(reflect)模式

例:

flags:

flag 作用
-source 包含要mock的接口文件
-destination 生成的源代码写入的文件,若不设置此项,代码将打印到标准输出
-package 用于生成的模拟类源代码的包名,若不设置此项,默认在原包名前添加mock_前缀
-imports 在生成的源代码中使用显式导入列表
-aux_files 需要参考以解决的附加文件列表
-build_flags (仅反射模式)传递标志给go build
-mock_names 生成的模拟的自定义名称列表
-self_package 生成的代码的完整包导入路径
-copyright_file 用于将版权标头添加到生成的源代码中的版权文件
-debug_parser 仅打印解析器结果
-exec_only (反射模式)如果设置,则执行此反射程序
-prog_only (反射模式)只生成反射程序,将其写入标准输出并退出
-write_package_comment 若为true,则写入包文档注释(godoc),默认为true

软件测试中的打桩是指用一些代码(桩stub)代替目标代码,通常用来屏蔽或补齐业务逻辑中的关键代码方便进行单元测试。

屏蔽:不想在单元测试中引入数据库连接等重要资源
补齐:依赖的上下游函数或方法还未实现
桩:或称桩代码，是指用来代替关联代码或者未实现代码的代码。如果函数B用B1来代替，那么，B称为原函数，B1称为桩函数。打桩就是编写或生成桩代码。

monkey
除了上述的gomock以及gostub工具可以进行mock接口以及打桩外,在go中还有一个更强大的打桩工具————monkey,它支持为任意函数及方法进行打桩,其在运行时通过汇编语言重写可执行文件,将目标函数或方法的实现跳转到桩实现,其原理类似热补丁,但需要注意以下事项:
·不支持内联函数,在测试时需要通过命令行参数-gcflags=-l关闭go语言的内联优化
·不是线程安全的,不能用于并发的单元测试中

内联优化
在go的逃逸分析中,我们知道栈分配内存会比堆分配要高效,所以我们就会希望对象尽可能被分配在栈上,而一个goroutine会有一个单独的栈,栈又会包含多个栈帧,栈帧是函数调用时在栈上为函数所分配的区域。但其实函数调用存在一定固定开销,例如维护帧指针寄存器BP、栈溢出检测等,而对于一些代码行比较少的函数,编译器倾向于将它们在编译期展开从而消除函数调用,这种行为就是内联。通俗来讲,内敛就是将一个函数调用原地展开替换成这个函数的实现,当该函数被多次调用就会被多次展开,会增加编译后二进制文件的大小,而非内联函数只会保存一份函数体的代码然后进行调用,所以在空间上一般来说使用内联函数会导致生成的可执行文件变大。
在程序代码中，想要禁止编译器内联优化很简单，在函数定义前一行添加//go:noinline即可。
